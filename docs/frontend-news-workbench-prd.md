# ViraX 图文资讯工作台 PRD

## 1. 文档目标

本文档用于定义 ViraX 下一阶段的前端 UI 迭代范围。

本轮先聚焦产品结构与交互设计，优先完成可点击、可演示、可用模拟数据驱动的前端界面，用于确认产品方向和页面信息架构是否符合预期。

本轮不以真实后端接入为目标，后端数据模型、抓取能力、API 字段细化将在前端方案确认后再继续设计。

## 2. 项目背景

当前项目的核心能力是：

- 博主录入
- 视频内容抓取
- 信息流展示

当前首页主要服务于视频类内容监控，但缺少对图文资讯类内容的收集、筛选和呈现。

实际业务中，团队还需要追踪以下类型的资讯：

- 微信公众号文章
- X/Twitter 上的图文帖子、长帖、行业观察内容
- 后续可能扩展的小红书图文内容

这类内容的特点是：

- 以图文、文章、外链为主
- 更偏资讯、行业观察、媒体分析
- 需要人工判断内容是否有效
- 需要在首页集中呈现给内部团队

## 3. 本轮目标

本轮目标不是直接打通所有采集链路，而是先把产品升级为一个结构清晰的内部工作台，并完成图文资讯业务线的前端 UI。

本轮目标包括：

- 将现有单页结构升级为左侧导航的多页面工作台
- 新增图文资讯页面，作为默认首页
- 保留现有信息流页面，用于展示视频类抓取内容
- 新增录入后台页面，用于资讯录入与维护
- 使用模拟数据完成页面展示与交互验证

## 4. 核心产品原则

- 图文资讯与视频内容是两条并行内容线，不混在同一个 feed 中
- 首页默认展示图文资讯，而不是视频 feed
- 自动抓取内容不能直接进入首页，必须经过人工确认
- 当前所有内部用户默认都可录入，暂不做权限区分
- 第一版优先做结构清晰、内容可演示、状态可理解的 UI

## 5. 页面结构

产品调整为左侧导航布局，包含三个一级页面：

```text
左侧导航
├─ 资讯页
├─ 信息流页
└─ 录入后台
```

### 5.1 资讯页

用途：

- 面向团队展示已经确认有效的热点资讯
- 作为产品默认首页

内容来源：

- 手工录入后被标记为“已入选”的资讯
- 自动抓取后经过人工筛选、被标记为“已入选”的资讯

### 5.2 信息流页

用途：

- 保留当前已有的视频/帖子内容浏览能力
- 承接原首页内容

内容类型：

- 现有 Instagram / TikTok / YouTube 抓取内容

### 5.3 录入后台

用途：

- 作为运营和内部成员的维护入口
- 用于录入、编辑、筛选图文资讯

本轮建议包含：

- 新增资讯
- 编辑资讯
- 删除资讯
- 修改资讯状态
- 查看资讯列表
- 为后续“X 定向作者管理”预留入口区域

## 6. 资讯页设计要求

### 6.1 展示模式

资讯页采用“混合流”结构：

- 页面顶部：重点资讯卡片区
- 页面下方：最新资讯列表区

推荐结构：

```text
资讯页
├─ 页面标题 / 描述 / 筛选区
├─ 重点资讯区（1-3 条大卡）
└─ 最新资讯列表（高密度列表）
```

### 6.2 重点资讯区

目的：

- 展示最值得团队优先关注的资讯
- 强化视觉层级

每张卡片建议展示：

- 标题
- 简短摘要
- 封面图
- 来源平台
- 作者名
- 发布时间
- 状态标签或来源标签

交互：

- 点击整张卡片跳转到原始链接

### 6.3 最新资讯列表

目的：

- 高密度展示更多已入选资讯
- 便于快速浏览和查阅

每条列表项建议展示：

- 标题
- 一到两行摘要
- 来源平台
- 作者名
- 发布时间
- 录入方式标签

交互：

- 点击标题或整项跳转原文

### 6.4 筛选能力

本轮前端应支持基础筛选 UI，先用本地模拟数据驱动：

- 来源平台筛选：`X`、`微信公众号`
- 时间筛选：如“全部 / 近 7 天 / 近 30 天”
- 可选状态筛选：默认只展示 `featured`

### 6.5 资讯跳转逻辑

资讯页中的卡片和列表项默认直接跳转外部原文链接。

第一版不做站内资讯详情页。

原因：

- 微信公众号内容本身就是外部阅读场景
- 站内详情页会引入正文解析、富文本结构、图片落地等额外复杂度
- 当前阶段主要目标是确认首页资讯呈现方式

## 7. 信息流页设计要求

信息流页延续当前产品的已有能力，但从首页迁移为独立页面。

建议：

- 保留现有 feed 结构和筛选逻辑
- 页面名称可以调整为更明确的“视频信息流”或“内容信息流”
- 风格上与资讯页保持同一套左侧导航和顶层布局

本轮重点不是重做信息流业务，而是完成结构迁移与视觉统一。

## 8. 录入后台设计要求

录入后台是图文资讯业务线的操作入口。

### 8.1 页面目标

- 让内部成员快速新增资讯
- 让成员可以管理资讯状态
- 让自动抓取产生的候选内容未来可以进入同一管理池

### 8.2 页面建议结构

```text
录入后台
├─ 页面标题 / 说明
├─ 新增资讯表单
├─ 资讯列表
└─ 来源管理预留区（本轮可先做占位）
```

### 8.3 新增资讯表单字段

本轮以前端模拟需求为主，建议表单包含以下字段：

- 标题
- 摘要
- 原文链接
- 封面图链接
- 来源平台：`wechat | x`
- 作者名
- 发布时间
- 录入方式：默认 `manual`
- 状态：默认 `pending`

可选增强字段：

- 备注
- 标签

### 8.4 资讯状态

前端先统一支持以下三种状态：

- `pending`：待筛选
- `featured`：已入选
- `ignored`：已忽略

状态语义：

- `pending`：已录入，但未决定是否在资讯页展示
- `featured`：已确认有效，应出现在资讯页
- `ignored`：已判断不需要展示

### 8.5 列表管理能力

录入后台的资讯列表建议支持：

- 查看基础信息
- 通过状态快速筛选
- 快速修改状态
- 编辑已有记录
- 删除记录

### 8.6 权限约束

当前平台默认内部使用，本轮不做角色权限区分。

当前规则：

- 所有已登录用户都可录入
- 所有已登录用户都可编辑和修改状态
- 数据层建议未来保留 `created_by`、`updated_by` 字段，但本轮前端只需体现“录入人”概念即可

## 9. 内容来源策略

### 9.1 本轮纳入范围

- `X`
- `微信公众号`

### 9.2 暂不纳入范围

- `小红书`

原因：

- 平台风控和抓取限制更强
- 当前业务上优先级较低
- 容易影响第一版需求收敛

### 9.3 来源接入方式

#### X

定位：

- 后续重点自动化来源

本轮前端要求：

- 支持作为资讯来源类型出现
- 支持录入后台录入 X 链接类内容
- 为未来“X 定向作者抓取”预留后台入口

后续待确认项：

- X API 实际可返回字段
- 定向作者跟踪方式
- 自动抓取结果如何进入候选资讯池

#### 微信公众号

定位：

- 当前以人工录入为主

本轮前端要求：

- 支持手工录入公众号文章
- 资讯卡片点击后直接跳转公众号原文链接

当前不做：

- 正文抓取
- 正文存储
- 站内详情展示

## 10. 推荐前端信息架构

建议将图文资讯业务线与现有视频内容业务线分离。

逻辑关系如下：

```text
手工录入资讯 ─────┐
                  ├─> 候选资讯池 ──> 人工筛选 ──> 资讯页展示
X 自动抓取资讯 ───┘

现有博主管理 ─────> 视频采集体系
现有视频帖子 ─────> 信息流页展示
```

这意味着前端层面不应把图文资讯直接并入现有视频 feed 组件，而应单独设计页面与卡片体系。

## 11. 前端模拟数据要求

本轮前端 UI 开发建议完全使用模拟数据，以便快速验证页面结构和内容呈现。

建议至少准备三类模拟数据：

### 11.1 资讯页数据

需要覆盖：

- 重点资讯卡片数据
- 列表资讯数据
- 不同来源平台数据
- 不同发布时间数据
- 不同录入方式标签

### 11.2 录入后台数据

需要覆盖：

- `pending`
- `featured`
- `ignored`

并能体现：

- 状态切换
- 编辑入口
- 删除入口

### 11.3 来源管理预留数据

可以只做静态占位，用于表达未来会有：

- X 定向关注作者名单
- 自动抓取状态
- 最近更新时间

## 12. 推荐视觉与布局方向

本轮界面应该强调“内部工具工作台”感，而不是纯内容社区 feed 感。

建议方向：

- 左侧固定导航
- 主区域分层清晰
- 资讯页突出信息层级和可读性
- 录入后台强调操作效率
- 保持和现有产品一致的基础设计语言，但整体结构更接近典型运营工具

## 13. 本轮不做

以下内容明确不纳入本轮前端范围：

- 真实后端数据接入
- X API 真正联调
- 微信公众号正文采集
- 小红书来源接入
- 用户权限系统
- 站内资讯详情页
- AI 自动摘要
- AI 自动评分或推荐排序
- 日报自动生成

## 14. 前端验收标准

当前阶段的验收重点是“设计和产品方向确认”，不是后端联调完成度。

前端完成后应满足：

- 有完整的左侧导航三页面结构
- 默认首页为资讯页
- 资讯页可基于模拟数据完整展示“混合流”
- 信息流页可承接现有 feed 页面内容
- 录入后台可演示新增、查看、状态切换、编辑入口、删除入口
- 整体页面结构不混乱，业务边界清晰
- 团队可以通过前端界面直观看到产品未来形态

## 15. 下一阶段工作

当前端方向确认后，再进入后端设计阶段。

下一阶段再讨论：

- 图文资讯的数据模型
- X API 字段映射
- 自动抓取任务如何入库
- 候选资讯池与首页精选资讯之间的状态流转
- 后台 API 与数据库表设计

## 16. 一句话结论

本轮迭代的本质，是把当前的“视频监控单页工具”升级为“图文资讯 + 视频信息流 + 录入后台”三位一体的内部工作台，并先通过前端模拟数据把产品形态定义清楚。
